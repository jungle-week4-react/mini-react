# Virtual DOM Diff 프로젝트 요구사항 정의서

## 1. 프로젝트 개요
본 프로젝트의 목적은 React의 핵심 개념인 Virtual DOM과 Diff 알고리즘을 TypeScript와 브라우저 DOM API만으로 직접 구현하고, 그 결과를 웹 페이지에서 검증할 수 있도록 하는 것이다.

본 프로젝트는 학습용 이론 설명보다 실제 구현을 우선하며, 구현된 핵심 로직은 설명 가능한 수준으로 구조화한다.

## 2. 목표
- 브라우저의 실제 DOM 정보를 읽어 Virtual DOM 트리로 변환한다.
- 두 개의 Virtual DOM 트리를 비교하여 변경된 부분만 추출하는 Diff 알고리즘을 구현한다.
- Diff 결과를 Patch 연산으로 실제 DOM에 반영한다.
- 상태 이력을 저장하고 뒤로가기/앞으로가기 기능을 제공한다.
- 사용자가 웹 페이지에서 직접 수정 및 검증할 수 있는 테스트 환경을 제공한다.

## 3. 기술 스택
- Language: TypeScript
- Markup: HTML5
- Style: CSS3
- Runtime: Browser DOM API
- Framework: 없음 (Vanilla)
- JSX: 사용하지 않음
- Build Tool: TypeScript Compiler (`tsc`)

## 4. 핵심 개념 정의
- Virtual DOM
  - 실제 DOM 구조를 비교 가능한 JavaScript 객체 트리로 표현한 자료구조
- Diff
  - 이전 Virtual DOM과 현재 Virtual DOM의 차이를 계산하는 알고리즘
- Patch
  - Diff 결과를 실제 DOM에 반영하기 위한 변경 명령 집합
- History
  - 각 시점의 Virtual DOM 스냅샷을 저장하고 undo/redo를 지원하는 상태 관리 구조

## 5. 기능 요구사항

### 5.1 초기화
- 페이지 로드 시 실제 영역의 샘플 HTML을 기준으로 초기 DOM 트리를 구성한다.
- 실제 영역의 DOM을 읽어 초기 Virtual DOM 트리를 생성한다.
- 생성된 Virtual DOM을 사용하여 테스트 영역을 렌더링한다.
- 초기 Virtual DOM은 History의 첫 번째 스냅샷으로 저장한다.

### 5.2 화면 구성
- 페이지에는 다음 요소가 반드시 존재해야 한다.
- 실제 영역
- 테스트 영역
- Patch 버튼
- 뒤로가기 버튼
- 앞으로가기 버튼

### 5.3 테스트 영역 수정
- 사용자는 테스트 영역의 HTML 구조를 자유롭게 수정할 수 있어야 한다.
- 테스트 영역은 브라우저 DOM으로 존재해야 하며, 수정 후 다시 Virtual DOM으로 변환 가능해야 한다.

### 5.4 Patch 동작
- Patch 버튼 클릭 시 테스트 영역의 현재 DOM 상태를 다시 Virtual DOM으로 변환한다.
- 이전 Virtual DOM과 현재 Virtual DOM을 비교하여 차이를 계산한다.
- 계산된 변경 사항만 실제 영역 DOM에 반영한다.
- 변경 후 새 Virtual DOM을 History에 저장한다.

### 5.5 Diff 규칙
- 노드 비교는 동일 위치 비교를 기본으로 하되, 형제 노드 비교에서는 `key`를 우선 사용한다.
- `key`가 있는 형제 노드는 같은 식별자로 간주한다.
- `key`가 없는 노드는 인덱스 기준으로 비교한다.
- 같은 형제 레벨에서 `key`는 유일해야 한다.

### 5.6 History 동작
- Patch 성공 시 새로운 Virtual DOM 스냅샷을 History에 추가한다.
- 뒤로가기 버튼 클릭 시 이전 스냅샷으로 이동한다.
- 앞으로가기 버튼 클릭 시 다음 스냅샷으로 이동한다.
- undo/redo 실행 시 실제 영역과 테스트 영역은 동일한 상태로 함께 갱신되어야 한다.
- undo 후 새 Patch가 발생하면 redo 이력은 제거한다.

## 6. 비기능 요구사항
- 코드 구조는 핵심 로직과 UI 로직이 분리되어야 한다.
- 주요 알고리즘에는 설명 가능한 수준의 주석을 포함해야 한다.
- 타입 정의를 통해 데이터 구조를 명확히 해야 한다.
- 프레임워크 없이 브라우저 표준 API만 사용해야 한다.
- 구현 결과는 발표 및 설명이 가능해야 한다.

## 7. 단계별 구현 범위

### 1단계. 트리 정의
- Virtual DOM 노드 구조를 정의한다.
- Patch 구조를 정의한다.
- History 상태 구조를 정의한다.
- `key` 사용 규칙을 정의한다.

### 2단계. 트리 생성 메서드
- 실제 DOM을 Virtual DOM으로 변환하는 메서드를 구현한다.
- Virtual DOM을 실제 DOM으로 복원하는 메서드를 구현한다.
- History 저장용 깊은 복사 메서드를 구현한다.

### 3단계. Diff 구현
- 이전 트리와 현재 트리를 비교하는 메서드를 구현한다.
- 텍스트, 속성, 노드 교체, 삽입, 삭제를 판별한다.
- 형제 노드는 `key` 기반으로 비교한다.

### 4단계. Patch 적용 및 DOM 관리
- Diff 결과를 실제 DOM에 반영하는 메서드를 구현한다.
- 실제 영역과 테스트 영역의 렌더링을 관리한다.
- 버튼 상태와 DOM 동기화를 관리한다.

### 5단계. 구조화
- History와 UI 이벤트를 연결한다.
- 엔트리 파일에서 초기화 흐름을 정리한다.
- 파일을 역할별로 분리한다.

## 8. 수용 기준
- 초기 로드 시 실제 영역 DOM이 Virtual DOM으로 정상 변환된다.
- 테스트 영역은 초기 Virtual DOM과 동일한 구조로 렌더링된다.
- 텍스트 변경 시 변경된 텍스트만 실제 영역에 반영된다.
- 속성 변경 시 변경된 속성만 실제 영역에 반영된다.
- 노드 추가/삭제 시 해당 노드만 실제 영역에 반영된다.
- 같은 형제 레벨에서 `key` 기반 비교가 동작한다.
- Patch 후 새 상태가 History에 저장된다.
- undo/redo 시 실제 영역과 테스트 영역이 동일 상태로 이동한다.
- redo 가능한 상태에서 undo 후 새 Patch를 수행하면 redo 이력이 제거된다.
