# 리액트의 핵심

리액트의 핵심은 생각보다 단순하다.

바닐라 JavaScript로 DOM을 직접 다루다 보면 계속 이런 일을 하게 된다.

- 어떤 노드를 찾아야 하는지 기억해야 한다.
- 텍스트를 바꿀지, 속성을 바꿀지, 자식을 추가할지 직접 판단해야 한다.
- 여러 상태가 동시에 바뀌면 어떤 순서로 DOM을 고쳐야 하는지도 직접 관리해야 한다.

즉, 화면이 바뀔 때마다 사람이 브라우저에게 "여기 바꾸고, 저기 지우고, 이건 새로 넣어"를 하나씩 지시해야 한다.
리액트는 이 귀찮은 과정을 대신 맡아준다.

## 왜 Virtual DOM이 필요한가

리액트가 DOM 변경을 대신 처리하려면, 먼저 화면을 표현할 수 있는 중간 형태가 필요하다.
그게 바로 **Virtual DOM**이다.

Virtual DOM은 실제 브라우저 DOM이 아니라, 현재 UI를 JavaScript 객체 트리로 표현한 것이다.

예를 들어 브라우저의 DOM이 아래와 같다면:

```html
<div id="app">
  <p>Hello</p>
</div>
```

Virtual DOM은 대략 이런 트리로 생각할 수 있다.

```ts
{
  type: 'element',
  tag: 'div',
  props: { id: 'app' },
  children: [
    {
      type: 'element',
      tag: 'p',
      props: {},
      children: [
        { type: 'text', value: 'Hello' },
      ],
    },
  ],
}
```

중요한 점은, 리액트가 실제 DOM을 바로 만지는 대신 먼저 이 트리를 다룬다는 것이다.

## 변경은 어떻게 처리하나

화면 상태가 바뀌면 리액트는 "다음 화면"에 해당하는 새로운 Virtual DOM 트리를 만든다.
그리고 이전 트리와 새로운 트리를 비교해서 무엇이 달라졌는지 찾는다.

이 비교 과정이 **Diff**다.

Diff를 통해 리액트는 다음과 같은 변경점을 계산할 수 있다.

- 텍스트가 바뀌었는가
- 속성이 바뀌었는가
- 노드가 추가되었는가
- 노드가 제거되었는가
- 아예 다른 노드로 교체되어야 하는가

핵심은 실제 DOM 전체를 매번 새로 그리는 것이 아니라, **변경된 부분만 찾아서 반영한다**는 점이다.

## 흐름으로 보면 더 단순하다

리액트의 핵심 흐름은 아래 네 단계로 요약할 수 있다.

1. 현재 화면을 Virtual DOM 트리로 표현한다.
2. 상태가 바뀌면 새로운 Virtual DOM 트리를 만든다.
3. 이전 트리와 새로운 트리를 Diff 한다.
4. 변경된 부분만 실제 브라우저 DOM에 반영한다.

즉, 개발자는 "어떤 화면이 되어야 하는가"를 선언하고, 실제 DOM 조작은 리액트가 계산해서 처리한다.

## 이 프로젝트에서의 의미

이 프로젝트는 바로 그 핵심을 직접 구현하는 연습이다.

- `createVNodeFromElement`: 브라우저의 실제 DOM을 읽어서 Virtual DOM으로 바꾼다.
- `diffVNode`: 이전 Virtual DOM과 다음 Virtual DOM을 비교해서 patch 목록을 만든다.

이 두 가지가 연결되면, "현재 화면을 트리로 읽고", "변경점을 계산하고", "필요한 부분만 반영하는" 리액트의 핵심 아이디어를 직접 확인할 수 있다.

## 정리

리액트의 핵심은 화려한 문법이 아니다.
컴포넌트, 훅, 상태 관리도 중요하지만, 렌더링 관점에서 보면 핵심 아이디어는 훨씬 단순하다.

- 실제 DOM을 직접 매번 조작하지 않는다.
- 화면을 Virtual DOM이라는 트리로 표현한다.
- 이전 트리와 다음 트리를 비교한다.
- 바뀐 부분만 실제 DOM에 반영한다.

한 줄로 정리하면 이렇다.

> 리액트는 "DOM 조작을 개발자 대신 계산해 주는 시스템"이고, 그 중심에는 Virtual DOM과 Diff가 있다.
